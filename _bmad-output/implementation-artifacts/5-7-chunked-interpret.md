# Story 5.7: Chunked/windowed real-LLM interpret (unblocks the deferred full-session bake)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the operator running the full-session `claude -p` bake of the Story 10.1 session,
I want the `--cli`/`--real` interpret step WINDOWED into bounded chunks (each well under the CLI timeout) whose per-chunk annotations are merged + deduped,
so that the whole ~689-event reduced tagging-view can be interpreted without exceeding `ClaudeCliClient`'s 600s `execFile` timeout (or hallucinating refs over a huge input), while `annotationHash`/freeze/provenance and the committed fixture bundle stay UNCHANGED. (FR-2, NFR-2, NFR-5)

**When** interpreted,

### §0 — Current state + the WHY (verified at HEAD)

- **The timeout problem (the WHY for AC1/AC4):** at code HEAD `37b06d6` (registered at `16fc207`), the `--cli`/`--real` bake does ONE `interpreter.interpret(taggingView, scrubResult.scrubbedEvents)` over the WHOLE reduced view (`scripts/build-bundle.ts:201`). A single `claude -p` over that volume (a) EXCEEDS `ClaudeCliClient` `DEFAULT_TIMEOUT_MS = 600_000` (`src/llm/claude-cli-client.ts:43`; `execFile`'s `{ timeout }` kills the child -> exit 143 / SIGTERM -> the fail-loud non-zero-exit guard at L135-144 throws), and (b) over a 600+-event input the model occasionally hallucinates eventIds (the very reason the existing `sanitizeAgainstEvents` trim exists — `claude-interpreter.ts:184-212`). The reduced view already SHRANK the prompt (Story 5.6) but the SESSION COUNT is still too large for ONE call. Chunking windows the prompt so each call is small + fast.

### §1 — Why a PURE orchestrator with an INJECTED interpret callback (the testability decision)

The orchestrator (`src/interpret/chunked-interpret.ts`) takes the `interpret` function as an INJECTED dependency (`(promptEvents, groundingEvents) => Promise<BeatAnnotation[]>`) rather than importing/constructing `ClaudeInterpreter`. **Decision:** dependency-injection, so the orchestrator is (a) PURE-of-the-SDK (it never imports `@anthropic-ai/sdk` — R4 proof by construction: `grep -ril anthropic dist/` stays empty, and the orchestrator can be unit-tested with a FAKE that records its calls — ZERO LLM spend, the Story 3.2/5.6 mocked-machinery precedent), and (b) gate-testable for chunk coverage/merge/dedup WITHOUT a real client. The script (Task 4) binds `(chunk, grounding) => interpreter.interpret(chunk, grounding)` as the callback, so the SDK/`claude -p` zone stays entirely in `claude-interpreter.ts` + `scripts/`. This mirrors the established injectable-client seam (`AnthropicLike` in `claude-interpreter.ts:26-30` + `claude-cli-client.ts`): inject the heavyweight dependency so the logic is testable. The orchestrator is in `src/interpret/` (NOT `src/bundle/`) because it is a Layer-1 INTERPRET concern (it orchestrates the BeatInterpreter), is R4-allowed (though it uses no SDK), and `src/interpret/ -> src/bundle/` is already a TYPE-only edge (`claude-interpreter.ts:3`). vitest runs `src/**/*.test.ts`, so it is gate-covered (the thin-script precedent — `scripts/build-bundle.ts` stays thin glue).

### §2 — The 2-arg interpret seam is reused AS-IS (provenance intact, AC2)

The Story 5.6 `interpret(promptEvents, groundingEvents = promptEvents)` seam is the EXACT contract this story needs and is reused with NO change. Per call:
- `promptEvents` = the CHUNK (small) is serialized into `messages[0].content` (`claude-interpreter.ts:139`) — so each `claude -p` request is bounded (AC4).
- `groundingEvents` = the FULL scrubbed events on EVERY call — `stampProvenance` filters it for `sourceHash` (`claude-interpreter.ts:218-229`), and `sanitizeAgainstEvents` checks the model's refs against the FULL id set (`claude-interpreter.ts:184-212`), so per-annotation `sourceHash` stays over the full events and any chunk-local hallucination is trimmed against the full authoritative set. **Decision:** pass `scrubResult.scrubbedEvents` (the full set) as `groundingEvents` on every chunk call — referentially the SAME array each time (the AC2 test asserts `===`). `assembleBundle`'s `annotationHash`/freeze run over `scrubResult` UNCHANGED. The `BeatInterpreter` interface stays single-arg (`beat-interpreter.ts:11-13`); the 2-arg form is a `ClaudeInterpreter`-private extension already present at HEAD — this story adds NO interpreter signature change.

### §3 — `MAX_CHUNK_EVENTS` + the AC4 size/timeout bound (the budget)

- **`MAX_CHUNK_EVENTS = 100`** — a documented module `const`, NOT config-as-data. Rationale: this is a FIXED prompt-budget engineering knob (how many reduced-view rows fit comfortably under the `claude -p` timeout per call), not operator-tunable replay tuning (NFR-4 governs replay config — translation/pacing/captions — not a bake-input budget). Mirrors `SNIPPET_MAX_CHARS` (Story 5.6 §5) and `claude-interpreter.ts`'s `MAX_TOKENS` (`claude-interpreter.ts:52`) — fixed engineering ceilings kept inline. A config loader would earn nothing for a single internal build-time budget.
- **AC4 bound (the WHY):** each reduced-view row is `<= ~280` bytes worst-case (Story 5.6 §5: eventId ~12 + toolName ~8 + role ~8 + outcome ~9 + snippet `<= 200` + JSON overhead ~40). 100 rows -> `~28KB` per chunk prompt. **Decision:** the AC4 test asserts each chunk's `Buffer.byteLength(JSON.stringify(chunk)) < 35_000` (a conservative bound covering the worst-case ~28KB + headroom) and `chunks.length === 7` for a 689-event view @ 100 (`ceil(689/100) = 7`), and that the WHOLE view serializes materially larger than any one chunk (~193KB, the one-shot size that timed out). ~28KB per `claude -p` call is far under the 600s timeout that the ~193KB whole-view one-shot exceeded — the timeout fix proven at the gate (the actual sub-timeout latency is operator-verified — gate proves the BYTE budget that drives it). The chars/4 token heuristic (Story 5.6 §5): ~28KB/4 ≈ ~7K tokens/chunk, far under the ~200K window AND fast enough to return well inside 600s.

### §4 — Merge + dedup: highest-confidence, stable, chunk-order (AC3)

Chunking can produce DUPLICATE beats: a beat whose grounding spans a chunk boundary, or the model emitting the same anchor beat from two adjacent chunks, yields two annotations with the same (`eventRef`, `beatType`). **Decision:** dedup KEY = `${eventRef} ${beatType}` (the ` ` joiner avoids concatenation collisions). Walk the concatenated (chunk-order) array; keep the HIGHEST-`confidence` annotation per key; EQUAL confidence keeps FIRST-SEEN (stable tiebreak — `>` not `>=`). Output preserves FIRST-SEEN insertion order via a `Map<string, BeatAnnotation>` (`Map` preserves insertion order; replace the value in place when a strictly-higher dup arrives; never re-insert). Rationale: (a) highest-confidence is the most defensible single annotation when the same beat is seen twice; (b) a stable, order-deterministic merge keeps the whole pipeline reproducible (rebuild byte-identical — a Hard Invariant); (c) dedup on (`eventRef`,`beatType`) NOT `groundingPointer` is intentional — two chunks may ground the same anchor to slightly different ref sets, and the highest-confidence one's grounding wins as the single source of truth (documented; the kept annotation's grounding still resolves against the full events, so freeze stays fail-loud-clean). Sequential per-chunk calls (not `Promise.all`) keep the concatenation order = chunk order = deterministic merge input.

### §5 — Why this is safe + surgical (the inert-at-replay proof)

This story touches the `--cli`/`--real` BAKE PROMPT SHAPE only. At replay nothing changes: the browser loads the same payload-free `projectedEvents` bundle, folds the same baked timeline, resolves the same abstracted grounding (Story 5.5). The chunked orchestrator + the one wiring line live entirely in the offline `scripts/` + `src/interpret/` path. `annotationHash`/freeze are over the full scrubbed events (UNCHANGED — `assembleBundle` gets `scrubResult` as-is). Per-annotation `sourceHash` stays over the full events (the 5.6 2-arg seam, reused AS-IS). The committed fixture bundle is byte-identical (the mocked `else` path never builds the view nor chunks). BOTH golden snapshots are byte-stable (no Layer-0 code added). R1/R4/R5 hold (the orchestrator is SDK-free, phaser-free, `src/interpret/`-resident, importing only `../bundle/tagging-view` + `../schema/*` TYPES — no R1/R4/R5 crossing). The Saga, `replay-bundle.ts`, `merge.ts`, `tagging-view.ts`, `session-ingest.ts`, `freeze.ts`, `assemble-bundle.ts`, the `BeatInterpreter` interface, and `eslint.config.ts` are ALL untouched.
