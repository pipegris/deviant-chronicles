# Story 1.5: Aggregate and pace Events into a golden BattleTimeline

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the builder,
I want trivial event bursts collapsed and significant beats weighted into a paced `BattleTimeline`, guarded by a golden snapshot,
so that the Replay reads as drama not noise (FR-12) and determinism is mechanically enforced (NFR-2).

## Acceptance Criteria

> Verbatim from `epics.md#Story-1.5`. Each AC is mapped to Tasks below.

1. **Pacing pass (Given the translated action stream, When the pacing pass (`scoreEvent` / `windowEvents` / `deriveBeats`) runs, Then):** trivial/repetitive bursts aggregate into a single Action or brief montage while significant beats get visual weight/dwell time, all per `config/pacing-weights.json` + `config/window-config.json` (no hardcoded constants).

2. **Purity (Given `pace/` and all Layer-0 modules, When they execute, Then):** they are pure — no `Date.now()`/`Math.random()`/`performance.now()`/network/file-I/O/global mutable state; time derives only from event timestamps/orderKey (R2).

3. **Golden snapshot + watchable length (Given the fixture Session, When the pacing pass produces a `BattleTimeline`, Then):** a committed Vitest golden snapshot matches byte-for-byte on re-run, and any nondeterministic change fails the snapshot; **And** the resulting timeline targets a watchable ~2–4 minute Replay length.

### What this story is (and is NOT)
- **IS:** the pure, declarative **Layer-0 Pace** stage — `TranslatedAction[]` in, a typed `BattleTimeline` out, driven by `src/config/pacing-weights.json` + `window-config.json`. It is the FINAL step (4) in the pipeline `pace(translate(ingest(raw)))` and the **Epic-1 capstone** — its golden snapshot is the end-to-end determinism anchor (NFR-2). [Source: architecture.md#Communication Patterns; #Decision Impact Analysis "(4) Pace → BattleTimeline (+ golden snapshot)"; epics.md#Story-1.5]
- **IS NOT:** the battle state machine (`model/`, Story 2.1 — HP/Resolve/Insight Gauge math) or playback (`model/playback.ts`, Story 2.2). Pace produces the ORDERED, weighted, dwelled `Beat[]`; the model later REDUCES over it. Do NOT compute Problem-Integrity/Resolve bars here — those deltas already live on the `TranslatedAction` (carried forward via `sourceEventIds` grounding) and are applied by `model/`. [Source: epics.md#Story-2.1; `src/schema/battle-timeline.ts` L37-45 `BattleState` is the model's output, not Pace's.]
- **IS NOT:** interpretation. Beats carry NO `beatType`/dramatic label (R1) — the "Shaman/Dispel/Summon" tagging is Layer-1 (`interpret/`, Epic 3) and must NEVER appear on a `Beat`. [Source: architecture.md#R1; `src/schema/battle-timeline.ts` L7-10.]
- **IS NOT:** a second parser. `pace/` consumes the in-memory `TranslatedAction[]`; it NEVER re-reads raw JSONL (R3). [Source: architecture.md#R3]

### Windowing / montage policy — RESOLVED
- **A window is discrete OR montage.** Discrete = one significant action (weight > `montageThresholdWeight`). Montage = a maximal run of ≥`minRunToCollapse` consecutive trivial actions (weight ≤ threshold) on the SAME `orderKey.streamId`. A significant action always ENDS a run (and starts its own discrete window). A trivial run shorter than `minRunToCollapse` is NOT collapsed (it stays as discrete single-action windows) — this prevents over-collapsing a lone idle and keeps a 1:1 beat for isolated filler. Rationale: FR-12 says "trivial/repetitive BURSTS" collapse — a burst is a RUN, not a single event; `minRunToCollapse` encodes "burst". [Source: PRD FR-12 L178 "a run of small reads … aggregated into a single readable Battle Action or a brief montage"; epics.md#Story-1.5 AC1; brainstorm Mapping #5 "wave-clearing = the 'lots of tiny fixes' cleanup montage"]
- **Montage representative `actionType`:** a montage Beat still needs ONE `actionType` (the schema requires it). **Decision: use the most-significant actionType in the run** (e.g. a run of `idle`+`scout` → `scout`; a pure-`idle` run → `idle`) — document this so the renderer (Epic 2) knows a montage's type is "the headline of the collapsed run", not arbitrary. A defensible alternative is a fixed `idle` for all montages; choose the most-significant-in-run because it preserves a hint of what was collapsed (a scout-heavy montage reads as scouting) while still being a single low-weight Beat. Record the choice + rationale in the completion notes. [Source: `src/schema/battle-timeline.ts` BeatSchema requires `actionType`; FR-12 "single readable Battle Action"]
- **Montage `weight`/`dwellMs` stay LOW:** a montage's aggregated weight must be ≤ a real significant beat's (so significant beats out-dwell it — AC1/AC3). **Decision: montage weight = the run's MAX trivial weight (capped at `montageThresholdWeight`)**, NOT the sum (summing a long idle run could exceed a real beat and invert the dwell ordering). Document the cap. [Source: epics.md#Story-1.5 AC1 "significant beats get visual weight/dwell time"; AC3 "watchable" rhythm]
- **Stream discipline:** a run never crosses `streamId` (a main-stream idle run and an `orchestrator`-stream journal-started are separate windows) — same boundary the translate engine enforces for strike resolution.
