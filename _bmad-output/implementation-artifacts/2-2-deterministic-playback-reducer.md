# Story 2.2: Deterministic playback reducer

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a viewer,
I want playback driven by a deterministic cursor over the timeline that emits immutable battle snapshots,
so that play/pause/restart/scrub/speed all produce the exact correct state. (FR-6)

## Acceptance Criteria

(Verbatim from epics.md#Story-2.2, L263-270.)

1. **Given** a `BattleTimeline` and a playback cursor **When** the reducer advances/seeks **Then** it emits immutable `BattleState` snapshots and scrubbing to any cursor position yields the same state as playing to that position (no path dependence).
2. **Given** repeated runs **When** the same Session is replayed **Then** the experience is identical (deterministic; no randomness altering outcomes) **And** the reducer is a small typed function with no heavy store (no Redux/Pinia/XState).

### The load-bearing inheritance from Story 2.1 (READ THIS FIRST)
Story 2.1 was deliberately designed to de-risk this story. Its `foldBattleState(timeline, cursor, tuning)` is a **genuinely path-independent pure fold** — proven by a committed test that, for EVERY cursor `0..beats.length`, asserts `foldBattleState(tl, cursor)` deep-equals re-folding from scratch (`battle-model.test.ts` L82-87). Three facts you INHERIT and build on:
- **SCRUB==PLAY is already true at the model level.** Because the state at any cursor is a pure function of `(timeline, cursor)`, seeking to N and stepping to N necessarily produce the same `BattleState`. Your job is to (a) make the reducer ALWAYS derive its snapshot via `foldBattleState` (never by mutating the previous snapshot), so the property survives at the playback level, and (b) PROVE it at the playback level (the AC1 headline test). Do NOT re-engineer path-independence — inherit it. [Source: src/model/battle-model.ts L116-133; Story 2.1 Dev Notes "making foldBattleState a pure reduce is the load-bearing decision that de-risks 2.2"]
- **`foldBattleState` already clamps the cursor to `[0, beats.length]`** and returns a `BattleStateSchema.parse(...)`-validated state. So your `withCursor` clamp is belt-and-suspenders for the *playback* cursor field; you still clamp the PlaybackState `cursor` yourself (so `state.cursor` is the truthful clamped value), but you can rely on `foldBattleState` to never index out of range. `cursor <= 0` → initial state; `cursor > length` → the full (victory) fold. [Source: src/model/battle-model.ts L127 `clamp(cursor, 0, timeline.beats.length)`]
- **The emitted `BattleState` carries its own `cursor`** (= number of beats folded). So after `withCursor(state, N)` you have the invariant `state.battleState.cursor === state.cursor` (both clamped equally). The unit/acceptance tests pin this. [Source: src/model/battle-model.ts L111 `cursor: state.cursor + 1`, L98-103 of battle-model.test.ts]
- **Victory latches at the completion beat (cursor `beats.length` = 10 in the fixture).** So `seek`/`tick` to/past the end yields `battleState.victory === true` with the Boss at `hp 0`, and staying at the end keeps it victorious. No extra victory handling here — it falls out of the fold. [Source: src/model/battle-model.ts L88-90; battle-model.test.ts L198-227]

### The `PlaybackState` shape — RESOLVED on merits
- **DECISION — `PlaybackState = { status: 'playing' | 'paused'; cursor: number; speed: number; battleState: BattleState }`.** Rationale, field by field:
  - `status: 'playing' | 'paused'` — a string-literal union (project convention: "Enums as string-literal unions … No numeric enums"). Two states are sufficient for v0.1 (the PRD's control set is play/pause/restart/scrub/speed — there is no "stopped" distinct from paused-at-0; `restart` covers reset). A `'finished'` status was considered and REJECTED: victory is already observable on `battleState.victory`, and adding a third status would duplicate that truth and complicate the `tick`/`play` branches with no UI requirement asking for it (YAGNI). [Source: architecture.md#Naming Patterns L248; prd.md#FR-6 L167-170]
  - `cursor: number` — the playback position (an integer beat index in `[0, beats.length]`). It mirrors the `BattleState.cursor` but lives ALSO on `PlaybackState` so the reducer can advance/seek without unpacking the snapshot, and so a `seek` target is expressed in the same unit. Kept in lockstep with `battleState.cursor` by always routing cursor changes through `withCursor`.
  - `speed: number` — the LOGICAL step multiplier (beats advanced per `tick`); see "Speed is a logical multiplier". Lives on the state (not only in actions) because the Story 2.5 loop sets it once via `setSpeed` and then pumps bare `tick`s; the reducer reads `state.speed` to know the stride.
  - `battleState: BattleState` — the immutable snapshot the RenderPort (2.3) renders. ALWAYS `foldBattleState(timeline, cursor)`. Carrying the snapshot ON the playback state (rather than recomputing it in the renderer) means the reducer is the single source of "what to draw", matching "Playback → render is snapshot-based: the playback reducer emits immutable BattleState snapshots". [Source: architecture.md#Communication Patterns L272-273; src/schema/battle-timeline.ts L29-30 "the immutable playback snapshot the model emits (Story 2.2) and the RenderPort consumes (2.3)"]
- **Why no Zod schema for `PlaybackState`.** The project's Zod rule is "All cross-stage data is typed + Zod-validated at boundaries (Ingest input, Interpreter output)"; internal hand-offs trust types. `PlaybackState` is transient in-memory playback/UI state — it never crosses a parse boundary, is never serialized into the ReplayBundle, and is never read from an untrusted source. So a plain `type PlaybackState = {...}` (PascalCase) is correct; a runtime schema would be ceremony with no boundary to guard. (The `BattleState` it WRAPS is already Zod-validated by `foldBattleState`'s closing `BattleStateSchema.parse` — the fail-closed boundary is inherited.) [Source: architecture.md#Format Patterns L262-264 "typed + Zod-validated at boundaries … internal hand-offs trust types"; #Data Architecture "Validation: Zod … at the Ingest boundary only" L163-164]

### The action set — RESOLVED on merits
- **DECISION — `PlaybackAction = { type:'play' } | { type:'pause' } | { type:'restart' } | { type:'seek'; cursor } | { type:'tick' } | { type:'setSpeed'; speed }`.** This is the minimal discriminated union that covers FR-6's full v0.1 control set (the PRD ASSUMPTION L170 pins it: "scrub + play/pause/restart + a fast-forward toggle is the full control set for v0.1"):
  - `play` / `pause` ↔ FR-6 "Play, pause … behave predictably."
  - `restart` ↔ FR-6 "restart … behave predictably." (resets POSITION to 0; pauses.)
  - `seek { cursor }` ↔ FR-6 "scrub to any point and the Arena renders the correct state for that position." Named `seek` (the imperative verb) rather than `scrub`; the UI control is the scrubber, the action it dispatches is `seek`.
  - `setSpeed { speed }` ↔ FR-6 "at least one speed control (e.g. normal / fast)." A numeric `speed` (not a boolean fast-toggle) so normal=1, fast=2 (or higher) all express through one action; the UI can expose just two buttons while the reducer stays general.
  - `tick` ↔ the engine primitive: "advance playback by one logical step" (scaled by `speed`). FR-6 has no UI control for `tick` — it is what the Story 2.5 real-time loop pumps on each frame/interval while `playing`. Including it HERE (rather than letting 2.5 reach into the cursor directly) keeps ALL cursor motion inside the one pure reducer, so SCRUB==PLAY and determinism are provable in this story without the renderer. [Source: prd.md#FR-6 L163-171; epics.md#Story-2.5 L324-326 "driven by the reducer cursor from Story 2.2"]
- **Discriminated union on `type` (the idiomatic reducer shape).** Each action is a plain serializable object; the reducer switches on `action.type`. A `default: return state` (or exhaustive switch) keeps an unknown/future action a no-op rather than a throw — consistent with the project's "fail-closed-to-default, never crash" runtime forgiveness. (With `strict` + an exhaustive union the default is unreachable for known actions, but keep it as the safety net.) [Source: architecture.md#Process Patterns "Unmapped/unknown … fail-closed-to-default, never crash mid-replay" L278-279]
- **REJECTED actions (YAGNI):** `stepBack` / `prev` (scrubbing handles arbitrary backward motion; a dedicated back-step is not in FR-6), `toggle` (the UI composes `play`/`pause` from current status; a toggle would push status-reading into the action layer), `jumpToVictory` / chapter markers (no such control in v0.1). Add none of these. [Source: prd.md#FR-6 L170 ASSUMPTION pinning the v0.1 control set; CLAUDE.md "Simplicity first … nothing speculative"]

### Why a factory closing over the timeline (`createPlaybackReducer(timeline)`) — RESOLVED on merits
- **DECISION — export a factory `createPlaybackReducer(timeline, tuning = MODEL_TUNING): (state, action) => state` rather than a bare `playbackReducer(timeline, state, action)` or carrying the timeline on every action.** Rationale:
  - The `BattleTimeline` is CONSTANT for a replay session (one bundle = one timeline). Closing over it once yields the canonical `(state, action) => state` reducer signature the architecture names ("a small typed reducer + snapshot emitter") and that Story 2.5's loop/controls can hold and call without re-passing the timeline each dispatch. [Source: architecture.md#Frontend Architecture L194]
  - Putting the timeline on every action would bloat each action object with a large non-serializable payload and break the "actions are small plain descriptors" idiom — and the timeline is not user intent, it is context. Threading it as a third reducer param (`reducer(timeline, state, action)`) is workable but diverges from the standard 2-arg reducer shape the AC implies ("a small typed function") and that 2.5 will wire to controls. The closure is the cleanest fit.
  - The factory also closes over `tuning` (default `MODEL_TUNING`) so a test can build a reducer over an in-memory tuning variant — mirroring how every Layer-0 pure function here takes its config as a default param (`pace(actions, weights = PACING_WEIGHTS)`, `foldBattleState(tl, cursor, tuning = MODEL_TUNING)`), proving NFR-4 tunability without touching committed config. [Source: src/pace/derive-beats.ts L21-25; src/model/battle-model.ts L122-126; Story 2.1 Dev Notes "each pure function takes the config as a default param"]
- **`initialPlaybackState(timeline, tuning = MODEL_TUNING)`** is a separate exported helper (parallel to `initialBattleState`) so callers get the t=0 PlaybackState without dispatching. The reducer does NOT need an "init" action (the initial state is constructed directly) — keep the action union focused on user/loop intents.

### Transition semantics — the rationale per branch
- **`withCursor(state, nextCursor)` (the shared cursor-mover):** clamp `nextCursor` to `[0, beats.length]`, then return `{ ...state, cursor: clamped, battleState: foldBattleState(timeline, clamped, tuning) }`. EVERY cursor-changing branch (`restart`, `seek`, `tick`) goes through this one helper, so there is exactly ONE place that clamps and exactly ONE place that emits a snapshot — no branch can forget to re-derive `battleState` (the bug that would break SCRUB==PLAY). [Source: src/model/battle-model.ts L122-133 (the fold it delegates to)]
- **`play`** sets `status:'playing'`, nothing else. Idempotent. Crucially does NOT move the cursor or auto-advance — advancing is `tick`'s job (the loop pumps ticks). "Play at the end" simply sets `playing`; the next `tick` clamps and is a no-op (the viewer sees the held victory frame). This is simpler and more predictable than treating play-at-end as a special error/restart. [Source: prd.md#FR-6 L168 "play … behave predictably"]
- **`pause`** sets `status:'paused'`, nothing else. Idempotent. Cursor/snapshot/speed untouched — the renderer just stops pumping ticks.
- **`restart`** → cursor 0, `status:'paused'`, snapshot = initial, `speed` PRESERVED. Decision: restart resets POSITION (and pauses so the viewer can re-press play), but NOT the speed knob — a viewer who set 2x and restarts almost certainly still wants 2x; resetting speed would be a surprising side effect. (If product later wants restart-to-normal-speed it is a one-line change, but nothing in FR-6 asks for it.) Pausing on restart matches "replay" semantics (you restart to the start, then choose to play) and is the least-surprising default. [Source: prd.md#FR-6 L168; UJ-1 "the option to replay" prd.md L46]
- **`seek { cursor }`** → `withCursor(state, cursor)`; `status` and `speed` PRESERVED. Scrubbing while playing keeps playing (the scrubber can be dragged mid-play and playback resumes from the new point) and scrubbing while paused stays paused — FR-6 only requires "renders the correct state for that position", which `withCursor` guarantees. This is the literal SCRUB action and the AC1 SCRUB==PLAY property is proven against it. [Source: prd.md#FR-6 L169; epics.md#Story-2.2 AC1 L263-265]
- **`tick`** → `withCursor(state, state.cursor + state.speed)`. Advances by the logical stride. Status-AGNOSTIC by design: it is a pure "advance by speed" primitive; gating advancement on `playing` is the CALLER's responsibility (Story 2.5 only pumps `tick` while playing). This keeps the reducer minimal and the property tests simple (a test can tick a paused reducer to walk the timeline). At/past the end the clamp pins cursor at `beats.length` and holds the victory snapshot — repeated ticks are no-ops. [Source: epics.md#Story-2.5 (the loop that pumps ticks) L312-326; src/model/battle-model.ts victory latch L88-90]
- **`setSpeed { speed }`** → clamp to integer ≥ 1 and store; cursor/snapshot unchanged. See next.

### Speed is a logical multiplier, NOT wall-clock — RESOLVED on merits
- **DECISION — `speed` is an INTEGER ≥ 1 number of beats advanced per `tick` (a logical step multiplier), clamped via `Math.max(1, Math.floor(speed))`.** Rationale:
  - **TIME-FREE (R2):** the reducer must be pure and clock-free. Expressing speed as "beats per tick" keeps ALL of speed's effect inside integer cursor arithmetic — no `Date.now()`, no elapsed-time scaling, no `dwellMs` math. `setSpeed(2)` means each `tick` advances TWO beats; the renderer (2.5) decides how often to fire a `tick` in wall-clock terms (e.g. faster `dwellMs` pacing). The wall-clock mapping is firmly OUTSIDE this Layer-0 module. [Source: architecture.md#R2 L229-232; story-specific guidance "speed is a logical step multiplier, no wall-clock/Date.now"]
  - **Integer, ≥ 1:** a logical step must be a whole number of beats (the cursor indexes beats) and must move forward at least one beat per tick (speed 0 would freeze playback even while "playing" — a confusing dead state; negative speed has no v0.1 meaning). Flooring + `max(1, …)` makes `setSpeed(2.7)→2`, `setSpeed(0)→1`, `setSpeed(-3)→1` — robust against any UI passing a slider float. This is the only "validation" speed needs (no Zod, no config key — speed is a runtime UI choice, not battle tuning, so it stays OUT of `model-tuning.json`).
  - **FR-6 "at least one speed control (normal/fast)":** normal=1, fast=2 (or 3/4) all expressible; the UI in 2.5 can expose just two buttons mapping to `setSpeed(1)` / `setSpeed(2)`. A boolean `fast` flag was REJECTED because a numeric speed generalizes for free and is what a scrub-speed slider or a 4x button would need later, with no extra reducer code. [Source: prd.md#FR-6 L170; epics.md#Story-2.5 L322 "a speed control (at least normal/fast)"]
- NOTE: speed scales `tick` granularity, so at high speed a `tick` can jump OVER the exact completion beat (e.g. speed 3 from cursor 9 → cursor 10 clamped, landing exactly; speed 3 from cursor 8 → cursor 11→clamped 10). The victory snapshot still lands because the END is clamped to `beats.length` and victory latches at/through the completion beat — you never skip the victory state, you can only land ON or PAST it (both show victory). SCRUB==PLAY is asserted with `speed:1` (one beat per tick) so the per-cursor equivalence is exact; the speed>1 path is covered by its own unit tests. [Source: src/model/battle-model.ts L122-133 clamp; L88-90 latch]

### Purity & determinism (R2) — the WHOLE POINT (AC2)
- **The reducer is a pure function of `(state, action)` + the closed-over `(timeline, tuning)`.** Identical inputs ALWAYS yield a byte-identical `PlaybackState` (the AC2 determinism test runs a sequence twice and asserts `JSON.stringify` equality). No randomness, no clock, no IO. The only "memory" is the `state` argument threaded through — function-local, not module-level. [Source: epics.md#Story-2.2 AC2 L267-269; architecture.md#R2 L229-232]
- **R2 — FORBIDDEN in `playback.ts`:** `Date.now()`, `Math.random()`, `performance.now()`, network, `fs`/file IO, module-level mutable state. The reduce accumulator (the `state` arg) is PURE working memory — R2 forbids MODULE-level mutable state, not the value threaded through a pure reducer. Mirror the verbatim R2 reasoning in `battle-model.ts` L13-21 / `translate.ts` L15-20. [Source: architecture.md#R2 L229-232; src/model/battle-model.ts L13-21]
- **Immutability:** every branch returns a FRESH `PlaybackState` (`{ ...state, … }` or `withCursor(...)`); never mutate `state`, `state.battleState`, or the closed-over `timeline`. `foldBattleState` already returns a fresh, independently-built `BattleState` each call (fresh `enemies` array — `battle-model.ts` L102-104), so the emitted snapshot is never aliased to a prior one. The no-mutation tests are the mechanical guard (the same discipline Story 2.1 proved for the fold). [Source: src/model/battle-model.ts L102-113; architecture.md#Communication Patterns L272-273]
- **No golden snapshot for playback** (like the model, unlike `pace/`). The determinism anchors are the SCRUB==PLAY every-cursor equality + the run-twice byte-identical test (Task 4). [Source: Story 2.1 Dev Notes "There is NO golden snapshot for the model"]

### Edge cases & fail-closed behavior
- **seek/tick past the end → terminal (victory) state, and STAYS.** Clamp to `beats.length`; `foldBattleState` folds the whole timeline → `victory:true`, Boss `hp:0`. Repeated ticks/seeks at the end are no-ops (cursor pinned, snapshot identical). [Source: src/model/battle-model.ts L127 clamp, L88-90 latch]
- **seek/restart to/below 0 → initial state.** Clamp to 0; snapshot = `foldBattleState(tl, 0)` = `initialBattleState` (full bars, gauge 0, `victory:false`). [Source: src/model/battle-model.ts L128-132]
- **play at the end** → `status:'playing'` but the next `tick` is a clamped no-op (held victory frame). NOT an error, NOT an auto-restart. [Source: prd.md#FR-6 L168 "behave predictably"]
- **empty timeline (`beats: []`)** → `initialPlaybackState` at cursor 0; `tick`/`seek` clamp to `[0,0]` so cursor stays 0 and the snapshot stays the initial state; no throw. The committed schema accepts an empty timeline and `pace([])` returns an empty-beats timeline, so the reducer must tolerate it. [Source: src/schema/battle-timeline.unit.test.ts empty timeline valid; src/model/battle-model.unit.test.ts empty-timeline edge]
- **unknown action** → `default: return state` (a no-op) — fail-closed-to-default, never throw. [Source: architecture.md#Process Patterns L278-279]
- **setSpeed garbage** → clamped to integer ≥ 1 (see "Speed is a logical multiplier"). Never throws.
