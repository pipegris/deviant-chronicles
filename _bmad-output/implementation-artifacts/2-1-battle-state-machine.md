## Story

As the builder,
I want a pure battle state machine that advances Problem Integrity, Resolve, the Insight Gauge, and live Enemy instances from the Battle Action stream,
so that the Arena has a reproducible logical state at every timeline position. (FR-5)

(Verbatim from epics.md#Story-2.1, L242-253.)

1. **Given** a `BattleTimeline` **When** Actions are consumed in timeline order **Then** Problem Integrity and Resolve bars update consistently and are reproducible for any given timeline position.

### CRITICAL: the bars are derived from `actionType` + `weight`, NOT from deltas
- **The asymmetry that drives this whole story:** `TranslatedAction` carries `resolveDelta` and `problemIntegrityDelta` (the actual signed magnitudes), but the `pace` stage **drops them** ΓÇö `BeatSchema`/`deriveBeats` keep ONLY `{ orderKey, actionType, sourceEventIds, weight, dwellMs }` (see `src/pace/derive-beats.ts` L43-55: the Beat is built with no delta fields). So by the time the model folds the timeline, the per-action deltas are GONE. The model therefore CANNOT "apply the deltas" ΓÇö it must RE-DERIVE bar changes from `beat.actionType` (the metaphor verb) + `beat.weight` (the significance), using config magnitudes. This is the central design constraint; do not look for deltas on a Beat ΓÇö they aren't there. [Source: src/pace/derive-beats.ts L43-55; src/schema/battle-timeline.ts BeatSchema L11-18; src/translate/translated-action.ts L11-26 (deltas exist HERE, upstream)]
- **The metaphor ΓåÆ mechanic mapping (the source of truth for which verb does what):** [Source: prd.md FR-5 acceptance L159-161; FR-3/Translation Map L131-133; brainstorming Mapping #3 L56]
  | Beat `actionType` | Real-world meaning | Battle-state effect |
  |---|---|---|
  | `melee` | Edit/Write strike, or a passing result that resolved a strike (Boss hit), or the journal completion (passing work) | **Damages Problem Integrity** (= Boss HP). Higher `weight` ΓçÆ bigger hit (a Boss-hit/mirage melee scores 16-20 vs a base 10). |
  | `spell` | `Bash` test/build channeled spell | **Damages Problem Integrity** (a cast that lands progress). |
  | `counter` | A failing test/result ΓÇö the enemy's counter-attack (the "true-duel" backfire) | **Drains Resolve** AND **charges the Insight Gauge** (this is the struggle signal). |
  | `summon` | `Task` sub-agent / ally entrance | v0.1: a no-op on bars (the cinematic summon is Epic 3). Do not damage bars on a bare summon. |
  | `scout` | Read/Grep/Glob reveal | No bar effect (scouting reveals ground truth; it doesn't hit). |
  | `aetherStorm` | Environmental hazard | A pause ΓÇö no bar effect in v0.1 (the Pacer already treats it as a pause, not a hit; honor that). |
  | `idle` | Thinking / filler / montage headline | No bar effect. |
  - **Why `weight` scales the magnitude (not a flat per-verb constant):** the Pacer already encoded "how big a deal was this" into `weight` (a Boss-hit/mirage melee = 16-20, a base melee = 10, a spell = 12, a counter+drain = 13 ΓÇö see `score-event.ts` L13-22 and the golden snapshot). Reusing `weight` as the damage scalar means the model inherits the Pacer's significance tiering for free (a decisive Boss-hit melee out-damages a glancing one), and a future config/translation tune flows through with zero model code change (NFR-4). The config provides the per-effect SCALAR (e.g. `integrityDamagePerWeight`), `weight` provides the per-beat magnitude. [Source: src/pace/score-event.ts L13-22; src/pace/__snapshots__/pace.test.ts.snap weights; architecture.md NFR-4 / Anti-Patterns "Hardcoding pacing constants" L299]

### Breakthrough-from-Layer-0 heuristic ΓÇö RESOLVED (the prompt asks me to decide on merits)
- **The problem:** AC2 (and FR-5 L160) require the Insight Gauge to DISCHARGE on a "breakthrough", but in v0.1 the authoritative breakthrough tag is an **Annotation Sidecar (Layer-1)** that does not exist yet ΓÇö the PRD explicitly lists it as a beat "that cannot be reliably auto-detected in v0.1" and is tagged by the LLM Interpreter in Epic 3 (Story 3.x). The model MUST NOT import `interpret/` (R1). So this story needs a DETERMINISTIC, Layer-0-only breakthrough detector as a stand-in. [Source: brainstorming "Annotation Sidecar ΓÇª breakthrough that cannot be reliably auto-detected in v0.1" L62; prd.md FR-4 L111; epics.md#Story-2.1 STORY-GUIDANCE "breakthrough detectable from LAYER-0 signals only ΓÇª LLM-tagged breakthrough is Layer-1 and arrives in Epic 3"]
- **DECISION ΓÇö breakthrough = a decisive Problem-Integrity-damaging beat that immediately follows a charged Insight Gauge (i.e. follows a struggle run).** Concretely, in `applyBeat`: when the beat damages Problem Integrity (`melee`/`spell`) AND the *incoming* `insightGauge` is at/above `insight.dischargeThreshold`, treat it as a breakthrough ΓåÆ discharge the gauge to 0 on that beat (and the integrity damage still lands). Tune `chargePerStruggle` + `dischargeThreshold` so the fixture's single `counter` (beat 7) charges the gauge to/over the threshold, and the next integrity strike (the completion `melee` at beat 10) discharges it. This makes the fixture's struggleΓåÆbreakthrough arc (a failing test, then the work completes) drive the gauge exactly as the metaphor intends ("thrashing is charging up; the hard-won fix lands the summon"). [Source: prd.md FR-5 L160; brainstorming Mechanic #15 "charges from struggle ΓÇª discharges on the breakthrough" L86; Idea #30 "Rise Again" L120]
- **Why THIS heuristic (vs alternatives):**
  - *vs "discharge on the completion event only":* too coarse ΓÇö a session with an early breakthrough before completion would never discharge mid-fight; and it conflates breakthrough with victory (two distinct concepts in FR-5: L160 gauge discharge Γëá L161 Boss defeat). The chosen "decisive strike after a struggle run" fires at the RIGHT dramatic moment and degenerates gracefully (if the only post-struggle strike IS the completion, it discharges there ΓÇö which is exactly the fixture's case).
  - *vs "discharge on a high-weight strike (weight ΓëÑ X) regardless of gauge":* fires even when nothing was struggled-for ΓÇö violates "easy wins earn no summon" (brainstorming Mechanic #15). Gating on `insightGauge ΓëÑ dischargeThreshold` enforces "earned".
  - *vs "charge on long-thinking / retry-count too":* `retryCount` and timing live on `NormalizedEvent`, not on the `Beat` (dropped at the pace boundary like the deltas). The only struggle signal that survives to the `Beat` layer is the `counter` actionType. So Layer-0-at-the-model means struggle == `counter`. This is the honest, available signal; richer struggle detection (retries/long-thinking) belongs upstream (could be folded into `weight`/a future Beat field) or in the Layer-1 tagger.
  - **This is explicitly a v0.1 deterministic stand-in.** Document in the code + completion notes that Epic 3 replaces it with the LLM-tagged breakthrough (consumed via the read-only Layer-1 overlay by the renderer/portal ΓÇö NEVER by this Layer-0 model, which keeps computing its own deterministic gauge). The two never conflict because L1 never feeds mechanics (R1). [Source: architecture.md#R1; epics.md#Epic-3 / Story 3.3 "Translate the three signature beats into Battle behaviors"]

### The FOLD ΓÇö purity & reproducibility (R2) ΓÇö this is the WHOLE POINT (AC1)
- **`foldBattleState(timeline, cursor)` is a pure `Array.reduce` from `initialBattleState` over `beats.slice(0, cursor)`.** The state at ANY position is therefore a pure function of `(timeline, cursor, tuning)` ΓÇö there is NO path dependence and NO hidden mutable accumulator. ### Edge cases & fail-closed behavior
- **Empty timeline (`beats: []`):** `foldBattleState` returns `initialBattleState` (full bars, gauge 0, Boss at full hp, `cursor: 0`, `victory: false`). [Source: src/schema/battle-timeline.ts L43; battle-timeline.unit.test.ts L60-62]
- **Unmapped actionType:** any `actionType` with no configured effect is a NO-OP on the bars (fail-closed-to-default ΓÇö never throw mid-fold). All 7 union members are known, but writing the switch as "known effects + default no-op" honors the architecture's runtime-forgiveness rule and survives a future union addition. [Source: architecture.md#Process Patterns "fail-closed-to-default, never crash mid-replay" L278-279]
