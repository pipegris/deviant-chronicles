# Story 2.4: Core combat animations

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a viewer,
I want the core Battle Actions animated distinctly and smoothly,
so that the fight reads as a fluid FF-style battle. (FR-7, NFR-1)

## Acceptance Criteria

(Verbatim from `_bmad-output/planning-artifacts/epics.md#Story-2.4`, L296-310.)

1. **Given** the Forgemaiden **When** Battle Actions play **Then** she has at least idle, forge-strike (melee), cast (spell), stagger, and **rise/recover** animations **And** a burst of consecutive edits renders as a **Hammer Flurry** (visibly faster multi-strike) distinct from a single forge-strike.
2. **Given** a failure/stagger followed by a retry **When** the rise animation plays **Then** it visibly coincides with the Insight Gauge charging further (struggle -> power, reads as defiance not defeat).
3. **Given** the enemies and environment **When** animated **Then** Boss vs Minion/imp render distinctly with hit/death animations, the bars and gauge animate with state changes, and the Aether Storm has a distinct environmental visual **And** frame pacing stays smooth (~60fps target, no perceptible jank) on a mid-range laptop browser.

### What the gate CAN prove vs what needs OPERATOR eyes (READ THIS — be honest)

The headless pipeline (`typecheck` / `lint` / `test` / `build`) **CAN** prove the *testable surface*: the **pure `animation-plan` layer** — which animation INTENTS fire for a given `(prevBattleState, nextBattleState, beatsAdvanced)` transition: Hammer-Flurry-vs-single-strike (a pure function of the Beat's `sourceEventIds` cardinality), stagger->rise correlation with the gauge increment, hit/death intents, Aether-Storm trigger, and bar/gauge `from->to` deltas. It can also prove the **adapter wiring** (a fake adapter records the emitted intents), the headless Phaser BOOT smoke (the scene plays an intent list without throwing under `Phaser.HEADLESS`), R5 lint, and `pnpm build`.

It **CANNOT** prove **motion quality / 60fps / "reads as a flurry" / no-jank / "looks like defiance"**. AC3's "~60fps, no perceptible jank" (NFR-1) is a **design target, OPERATOR-verified** by running `pnpm dev` and watching — it is NOT a headless-provable gate. AC1's "visibly faster multi-strike", AC2's "reads as defiance not defeat", and AC3's "distinct environmental visual" are likewise the *visual reading* of intents the gate cannot see. **Do NOT claim motion fidelity the tests cannot show.** In the Completion Notes, state exactly what the pure intent tests cover vs what needs human eyes. ### Hammer Flurry — the PURE detection heuristic, RESOLVED on documented merits
- **DECISION — `isHammerFlurry(beat) = beat.actionType === 'melee' && beat.sourceEventIds.length > 1`.** The Pacer's `windowEvents` collapses a run of consecutive TRIVIAL same-stream actions into ONE montage window carrying ALL the collapsed `sourceEventIds`, headlined by the run's most-significant `actionType` (`derive-beats.ts` then copies `sourceEventIds` onto the Beat verbatim). So a melee Beat whose `sourceEventIds.length > 1` IS, by construction, a montage of consecutive edits the Pacer judged a burst — exactly "a burst of consecutive edits" (AC1). A single edit that scored significant stays a discrete window with ONE source id -> a single `forge-strike`. The Beat's own comment makes this the intended hook: "montage-vs-discrete is encoded solely by `sourceEventIds` cardinality + weight/dwellMs, never a label — R1." This is a PURE function of the Beat (no Phaser, no state), so it is fully unit-testable — the headline gate-provable claim of this story. [Source: src/pace/window-events.ts L60-93 (montageWindow collapses a run, carries all ids, headlines the max actionType); src/pace/derive-beats.ts L39-55 (Beat.sourceEventIds = window.sourceEventIds); story-specific guidance "the flurry-vs-single DECISION is a PURE function of the Beat (e.g. sourceEventIds.length / weight)"]

The flurry intent carries `repeat = sourceEventIds.length` so the Phaser layer fires that many quick sub-strikes (the "multi-strike"). [Source: src/pace/window-events.ts L74-93 (montage weight is CAPPED at the threshold, NOT summed — so weight does not grow with the burst length; cardinality is the burst-size signal); story-specific guidance "sourceEventIds.length / weight"]

### stagger -> rise <-> Insight Gauge — the testable correlation, RESOLVED on merits
- **DECISION — rise fires when an integrity-progress strike (`melee`/`spell`) plays in a transition where `next.insightGauge > prev.insightGauge`.** The Battle Model charges the Insight Gauge by `chargePerStruggle` on EVERY `counter` (struggle) beat and discharges it to 0 only on a breakthrough strike (gauge >= dischargeThreshold). So the gauge climbing (`next > prev`) means a struggle just charged it; a Forgemaiden strike landing as the gauge climbs is the "retry after a stagger, gauge charging further" beat AC2 describes (struggle -> power, defiance not defeat). Emitting `rise` + a `gauge-tween (from<to)` TOGETHER makes the rise visibly COINCIDE with the gauge charging — the testable correlation. [Source: src/model/battle-model.ts L82-97 (chargePerStruggle on counter; discharge on breakthrough strike); epics.md#Story-2.4 AC2 L304-305; story-specific guidance "the rise animation coincides with the Insight Gauge charging (struggle->power)"]
- **The stagger itself** is the `counter` beat: `counter` drains Resolve AND charges the gauge, so a `counter` transition shows `next.resolve < prev.resolve` and `next.insightGauge > prev.insightGauge`. Emit `stagger` (the recoil) + the `gauge-tween` on that transition. The subsequent `rise` is on the NEXT strike transition. This is a 2-transition arc (stagger, then rise-on-retry), both detectable from the deltas — no cross-frame state machine needed in the PURE layer (each `planAnimations` call sees one transition; the "followed by a retry" is simply the next call where the gauge is still climbing on a strike). [Source: src/model/battle-model.ts L91-97 (counter: Resolve down + gauge up); epics.md#Story-2.4 AC2 L304]
- **Breakthrough is NOT a rise.** When `prev.insightGauge >= dischargeThreshold` and a strike lands, the model discharges the gauge to 0 (`next.insightGauge < prev.insightGauge` — it goes DOWN). That is the breakthrough/discharge (the THUNDORR moment, Epic 3), NOT a rise. Emit a `gauge-tween` to 0 (the discharge animates) but NO `rise`. Keying rise on `next > prev` (a CHARGE) cleanly excludes the discharge. [Source: src/model/battle-model.ts L82-84 (discharge to 0 on breakthrough); epics.md#Story-3.3 L386-389 (the breakthrough/Eidolon Summon is Epic 3)]
- **What the fixture exercises:** the committed fixture has a `counter` beat at index 6 (a genuine stagger: Resolve down, gauge up), preceded/followed by melee/spell strikes — so the stagger and the gauge-charge tween are exercised on REAL data. Drive the rise/stagger tests from the fixture walk where possible; hand-build only the breakthrough/discharge branch if the fixture's gauge never reaches `dischargeThreshold` (check the folded gauge values in the walk). [Source: src/pace/__snapshots__/pace.test.ts.snap (counter beat at index 6); src/render/render-model.test.ts L43-57 (the fixture walk helper)]
