# Story 3.2: Real frozen LLM Interpreter baked into the bundle

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the builder,
I want an offline authoring script that runs the real LLM interpreter and freezes + hashes its annotations into the ReplayBundle,
so that interpretation is reproducible once frozen and never runs at replay time. (FR-2, NFR-2, NFR-5)

## Acceptance Criteria

(Verbatim from `_bmad-output/planning-artifacts/epics.md#Story-3.2`, under "### Story 3.2: Real frozen LLM Interpreter baked into the bundle".)

1. **Given** `scripts/interpret.ts` **When** it runs offline **Then** it calls `claude-sonnet-4-6` via structured/tool output, Zod-validates the result into `BeatAnnotation[]`, and `@anthropic-ai/sdk` is imported only here/`interpret/` (never browser-reachable; no API key in client) (R4).
2. **Given** the produced annotations **When** frozen **Then** they are content-addressed via `annotationHash = sha256(normalizedEvents + interpreterVersion + promptVersion)` and baked into the `ReplayBundle`; re-running with identical inputs yields the same hash.
3. **Given** interpreter quality concerns **When** evaluated **Then** `scripts/eval-interpreter.ts` measures tagging quality out-of-band and never gates CI; escalation to `claude-opus-4-8` is available if quality needs it.

### What the gate CAN prove vs. what is a deferred operator step (READ THIS FIRST)

This story builds the real-interpreter MACHINERY and tests **every** verifiable behavior against a **MOCKED** Anthropic client — **zero real network calls, no real `ANTHROPIC_API_KEY`** in dev or CI. The ONE real `claude-sonnet-4-6` annotation bake is a **deferred operator step** (Epic 5, Story 5.2 bundle assembly) and is explicitly NOT part of this story's gate. The FixtureInterpreter (3.1) remains the CI double for all downstream Layer-1 tests.

**Gate-provable here (typecheck / lint / test / build):**
- **AC1 (the SDK → structured-output → Zod pipeline)** — proven with a **fake Anthropic client injected** into `ClaudeInterpreter`: a fake whose `messages.create(...)` returns a canned forced-tool `tool_use` block → the interpreter maps + `BeatAnnotationSchema.parse`-validates it into `BeatAnnotation[]`. Assert: the request was built with `model: 'claude-sonnet-4-6'`, `tool_choice: { type: 'tool', name: '<tool>' }`, and a tool whose `input_schema` carries the annotation shape; the returned array is schema-valid; a malformed tool payload throws (fail-loud at the Layer-1 boundary). **R4 isolation** is proven by `grep -ril anthropic dist/` finding NOTHING (the SDK is import-only in `interpret/claude-interpreter.ts` + `scripts/`, neither browser-reachable) + a source-grep that the freeze/hash module imports no SDK.
- **AC2 (freeze + content-address)** — proven by unit-testing the PURE `freezeAnnotations` / `annotationHash` functions in `src/interpret/freeze.ts`: stable key order (canonical JSON), identical inputs → identical hash (determinism), and a changed event / `interpreterVersion` / `promptVersion` → a DIFFERENT hash. A frozen-then-reparsed set is byte-stable (NFR-2) and each element `BeatAnnotationSchema.parse`-validates with resolving `groundingPointer`s.
- **AC3 (eval is out-of-band)** — proven by unit-testing the PURE `evaluateTagging` scorer in `src/interpret/eval.ts` (computes a tagging-quality metric over expected-vs-actual) AND asserting it is NOT wired into the CI gate: `scripts/eval-interpreter.ts` is a thin CLI under `scripts/` (which `vitest.config.ts`'s `include: ['src/**/*.test.ts']` does NOT run), and no `package.json` `test`/CI path invokes it. Escalation to `claude-opus-4-8` is proven as a typed config option on `ClaudeInterpreter`.

**Deferred operator step (NOT a gate — do NOT attempt in this story):** the single real `claude-sonnet-4-6` (or escalated `claude-opus-4-8`) call over the real Story 10.1 session, run by an operator with a real key, producing the committed frozen artifact. This story leaves a documented CLI entrypoint + the existing `.env.example` `ANTHROPIC_API_KEY` note for that future bake.

There is nothing here a human must eyeball (no pixels). If a test "can't" prove something, the seam is wrong — fix the seam (make the client injectable / the logic pure), not the test. ### The structured-output mechanism: FORCED tool_choice (not `output_config.format`) — resolved on documented merits

The architecture says "structured/tool output → Zod-validated `BeatAnnotation[]`" (L185). Two SDK mechanisms exist on 0.104.1: (a) **forced tool use** — declare a tool whose `input_schema` is the desired JSON Schema and set `tool_choice: { type: 'tool', name }` so claude MUST emit that tool, then read `block.input`; (b) the newer `output_config.format` JSON-schema mode (`messages.parse()`). **This story uses (a) forced tool_choice** because: it is the literal "tool output" the architecture names; it is the most universally-supported, mockable seam (the fake just returns a `tool_use` block — exactly the wire shape); and it does not depend on the structured-outputs model-support matrix or the `.parse()` helper's Zod coupling. The tool — call it `emit_beat_annotations` — has `input_schema = { type:'object', properties:{ annotations:{ type:'array', items:{ <BeatAnnotation JSON shape> } } }, required:['annotations'] }`. The SDK returns `block.input` ALREADY PARSED (an object, not a string) — do NOT `JSON.parse` it; pass `block.input.annotations` straight into `BeatAnnotationSchema.parse(...)` per element. (claude-api skill caveat "parse tool inputs with JSON.parse, never raw-string-match" applies to the SERIALIZED input on the wire; the SDK hands you the parsed object — Zod-validate that.) [Source: claude-api skill #tool-use-concepts "Tool Choice {type:'tool', name} → must use the specified tool", "input_schema is JSON Schema", "always parse with JSON.parse — never raw-string-match the serialized input"; architecture.md#LLM Integration L185 "structured/tool output → Zod-validated BeatAnnotation[]"; CLAUDE.md "Push back when warranted … surface tradeoffs" — the (a) vs (b) choice is surfaced here with rationale]

### The hash inputs: `normalizedEvents + interpreterVersion + promptVersion` (NOT the annotations) — resolved

`annotationHash = sha256(normalizedEvents + interpreterVersion + promptVersion)` (architecture L170-172, restated verbatim on the `ReplayBundle.annotationHash` field comment, `replay-bundle.ts` L26-28). The content address is over the INPUTS to the interpretation (the events + the two versions), NOT over the produced annotations. Rationale (architecture's "the ReplayBundle is the cache key, not the raw transcript", L172): given the SAME events + SAME interpreter version + SAME prompt version, the frozen interpretation is the canonical artifact for that triple — the hash identifies "which interpretation run produced this", so re-running the bake with identical inputs yields the identical hash (AC2's "re-running with identical inputs yields the same hash"). `canonicalJSON` (recursively sorted object keys, preserved array order) makes the serialization deterministic so the sha256 is stable across property insertion order. Computed via `node:crypto` `createHash('sha256').update(canonicalJSON({normalizedEvents, interpreterVersion, promptVersion})).digest('hex')`. [Source: architecture.md#Data Architecture L170-172, #Format Patterns L264-266 "SHA-256 over canonical JSON (stable key order)"; src/schema/replay-bundle.ts L26-28; epics.md#Story-3.2 AC2]
