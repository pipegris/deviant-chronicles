# Story 5.6: Reduced tagging view + multi-stream bundle merge (enables the full-session real bake)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the operator preparing the REAL `claude -p` bake of the full Story 10.1 session,
I want (1) a pure reduced "tagging view" that maps the full normalized session to a compact, payload-free-but-snippet-bearing per-event view for the interpreter/Saga bake INPUT, and (2) the bundle build to ingest the WHOLE session (dev + fix transcripts + the journal) by merging multiple transcript streams,
so that the ~689-event real session fits well under the model context window for a one-shot `claude -p interpret` while the SHIPPED bundle, `annotationHash`/freeze provenance, and the committed fixture bundle are all UNCHANGED. (FR-2, NFR-2, NFR-5)

## Acceptance Criteria

_Author's note: this story is a BUILD-SURFACED enablement story for the deferred full-session real bake; it is NOT in `epics.md` (there is no epics.md anchor — the ACs live in the operator's createBrief). The Given/When/Then ACs below are the authoritative source for this story, authored verbatim per the operator. They extend the Epic-5 bake machinery (`epics.md#Story-5.2`/`#Story-3.2`; `architecture.md#LLM Integration` L181-190) and the Story 5.5 payload-free posture — but they touch BAKE INPUT ONLY; nothing about the shipped bundle changes._

**AC1 — reduced tagging view (the size fix).**
**Given** the full normalized session,
**When** the bake prepares the interpreter/Saga input,
**Then** a PURE builder produces a compact per-event view `{eventId, toolName, role, outcome, bounded snippet (<=~200 chars, scrub-safe)}` that preserves `eventId`, carries NO full payload, and brings the whole 689-event session well under the context window; deterministic (same input -> same view, no mutation).

**AC2 — multi-stream merge (the input fix).**
**Given** dev + fix transcripts + the journal,
**When** `build-bundle` ingests with multiple transcript streams (repeated `--transcript`/`--stream-id` or a `--session-manifest`) + `--journal`,
**Then** it parses+normalizes each stream with its stream-id and `mergeStreams` them into one `orderKey`-total-ordered `NormalizedEvent[]`; a SINGLE `--transcript` still works (backward-compatible) and `pnpm bundle:story-10-1` produces the byte-identical committed fixture bundle.

**AC3 — bake over reduced view, provenance intact.**
**Given** the `--cli`/`--real` bake,
**When** the interpreter + Saga run,
**Then** they receive the REDUCED view (fits context) yet `annotationHash` + freeze are still computed over the FULL scrubbed events (provenance unchanged), and the produced annotations ground to real `eventId`s present in the reduced view.

**AC4 — context fit.**
**Given** the full ~689-event session,
**When** reduced,
**Then** the interpreter input size is bounded and well under the window (assert a concrete bound, e.g. reduced-view serialized size < ~250KB / token estimate < ~60K), proving a one-shot `claude -p interpret` now fits.

### Scope fence (READ FIRST — what this story IS and IS NOT)

- **IS:** bake-INPUT enablement. It adds (1) a PURE reduced-view builder under `src/bundle/` that maps `NormalizedEvent[]` → a compact `TaggingViewEvent[]` carrying a bounded scrub-safe snippet per event, reusing the Story 5.5 `classifyRole` for `role`; (2) multi-transcript ingest in `scripts/build-bundle.ts` (repeated `--transcript`/`--stream-id` pairs AND a `--session-manifest` JSON), merging `[...allTranscripts, journal]` via the EXISTING `mergeStreams`; (3) wiring so the `--cli`/`--real` interpreter + Saga are handed the REDUCED view (not the full events) while `freezeAnnotations`/`annotationHash` stay over the FULL scrubbed events. It builds the MACHINERY; the operator runs the real full-session bake afterward.
- **IS NOT:** a change to the SHIPPED bundle (the bundle stays the Story 5.5 payload-free `projectedEvents` projection — `replay-bundle.ts` is UNTOUCHED), a change to `annotationHash`/freeze/the content address (provenance stays over the full scrubbed events, AC3), a Layer-0 change (ingest/translate/pace/model untouched; BOTH golden snapshots stay byte-stable), a real bake run (this story does NOT call a billed/real LLM and does NOT regenerate the committed bundle with rich content), or a change to `src/ingest/merge.ts` (it already accepts `NormalizedEvent[][]` — §3). It does NOT touch Stories 5-3 (art) / 5-4 (deploy) — those remain backlog deferred operator/external steps.
- **HARD INVARIANTS (load-bearing, do NOT cross):**
  1. **The committed fixture bundle stays BYTE-IDENTICAL.** `public/bundles/story-10-1.json` is regenerated by `pnpm bundle:story-10-1` (single `--transcript`, mocked path) and must come out byte-for-byte identical to HEAD. The multi-stream extension is purely ADDITIVE; the single-`--transcript` path is unchanged. `committed-bundle.test.ts` + `committed-bundle.payload-free.test.ts` stay GREEN and are NOT weakened.
  2. **`annotationHash` is computed over the FULL scrubbed events** (AC3). The reduced view is the LLM PROMPT INPUT ONLY; it is NEVER hashed, NEVER shipped, NEVER the freeze input. `freeze.ts`/`assemble-bundle.ts` are UNCHANGED in what they hash.
  3. **The reduced view is scrub-safe.** The bake runs over SCRUBBED events (Story 5.1), so snippets are extracted from already-scrubbed `payload` (secrets/PII already redacted). The builder runs AFTER scrub — never on raw events.
  4. **The default (mocked) dev/CI path is unaffected.** The `FixtureInterpreter` + placeholder Saga path (no `--cli`/`--real`) does NOT use the reduced view (it returns the committed fixture beats content-independently). Only the `--cli`/`--real` branch is rewired. CI never bakes.

Every consequential decision below is RESOLVED on documented merits (this is autonomous — decide and proceed). Cited paths are to files read during analysis. **The load-bearing line: the reduced view is BAKE-INPUT ONLY — nothing about the shipped bundle, `annotationHash`/freeze provenance, or the committed fixture bundle changes.**

### §0 — Current state + the WHY (verified)

- **The size problem (the WHY for AC1/AC4):** the bake serializes the full `NormalizedEvent[]` straight into the LLM prompt — `ClaudeInterpreter.interpret` does `messages: [{ role: 'user', content: JSON.stringify(events) }]` (`claude-interpreter.ts` L114) and `SagaAuthor.authorSaga` does the same (`saga-author.ts` L96). For the THIN committed fixture (14 events) that is fine, but the REAL Story 10.1 session is ~689 events with full payloads (file contents, command bodies, tool-output) — ~294K tokens, which EXCEEDS the ~200K context window for a one-shot `claude -p interpret`. The reduced view shrinks the prompt to a compact per-event tag-salient row, targeting < ~60K tokens (§5).
- **The input problem (the WHY for AC2):** the bundle build currently ingests ONE `--transcript` + ONE `--journal` and `mergeStreams([transcript, journal])` (`build-bundle.ts` L108-117). The full session is dev + fix transcripts + the journal — multiple transcript streams. AC2 extends ingest to accept multiple transcripts (repeated flags or a manifest) and merge `[...allTranscripts, journal]`.
- **What must NOT move (verified):** the SHIPPED bundle is the Story 5.5 payload-free `projectedEvents` projection (`replay-bundle.ts` L55-90, `assemble-bundle.ts` L81-83) — UNTOUCHED here. `annotationHash = sha256(normalizedEvents + interpreterVersion + promptVersion)` over the FULL scrubbed events (`freeze.ts` L68-79, called from `assemble-bundle.ts` L74-79) — UNCHANGED. The committed `public/bundles/story-10-1.json` (13KB, 14 projectedEvents, 2 annotations) — byte-identical after this story.

### §1 — The reduced view PRESERVES `eventId` (the grounding contract)

The interpreter grounds every annotation back to `eventId`s (`eventRef` + `groundingPointer.eventRefs`), and freeze/assemble assert those refs resolve (`freeze.ts` L115-124; `assemble-bundle.ts` L88-117). `eventId` is the opaque, ref-stable id (`u-0001`, `u-0002#1`, `phase-dev-...#started`) the timeline/annotation/portal/freeze ALL key on (Story 5.5 Decision §1, confirmed). So the reduced view MUST carry `eventId` verbatim — if the LLM sees `u-0002#1` in the reduced prompt, it grounds on `u-0002#1`, and that id is present in BOTH the reduced view AND the full events (the freeze input), so the grounding resolves on both sides. This is the whole reason the view is per-event and id-keyed rather than a free-text summary. **Decision:** `TaggingViewEvent.eventId` is the normalized `event.eventId`, copied 1:1, same order — asserted in the test.

### §2 — `role` + `outcome` reuse Story 5.5, byte-for-byte identical derivation

The reduced view's `role` and `outcome` are derived EXACTLY as the shipped projection derives them: `role = classifyRole(resolveTargetPath(event))` and `outcome = event.isError ? 'isError' : 'success'` (`project-events.ts` L43-52). **Decision:** import `classifyRole` + `AbstractedRole` from `./classify-role` and `Outcome` from `../schema/replay-bundle`, and re-derive `resolveTargetPath` the SAME way `project-events.ts` does (it is file-local there; re-derive minimally OR — cleaner — export `resolveTargetPath` + the outcome helper from `project-events.ts` and reuse, since the view and the projection should agree by construction; prefer EXPORT-AND-REUSE to avoid drift between two derivations, and document). Rationale: the view and the projection are siblings describing the same events at the same abstraction; sharing the derivation guarantees `role`/`outcome` agree and gives one place to change. The outcome nuance (a `tool_use` is `isError:false`; the paired `tool_result` carries the real pass/fail) is INHERITED from 5.5 (§4 there) — honest at the per-event level; the SNIPPET (§6) is what surfaces the result's pass/fail head for the LLM.

### §3 — `mergeStreams` already supports multi-stream (NO merge change)

`mergeStreams(streams: NormalizedEvent[][])` (`merge.ts` L26) flattens ALL streams, sorts by the composite `orderKey` (logicalClock, streamId, seqWithinStream) — a TOTAL comparator (L9-15) — and rewrites `logicalClock` to a dense 0..n-1 merged index (L30-37), preserving `streamId`/`seqWithinStream` as origin provenance. AC2's `mergeStreams([...allTranscripts, journal])` is therefore a DIRECT call with more array elements — **no change to `merge.ts`.** The only new work is (a) ingesting N transcripts (Task 2) and (b) anchoring the journal after the max transcript timestamp (generalize `devMaxEpoch`). **Decision:** do NOT touch `src/ingest/merge.ts` (Layer-0, golden-snapshot-guarded). The per-stream `streamId` distinguishes dev vs fix in the total order; the journal anchors between/after phases via `normalizeJournal(records, devMaxEpoch + 1)` (`normalize.ts` L194-235), generalized to the global max across all transcripts.

### §4 — Testable logic under `src/`, thin glue in `scripts/` (the precedent)

Vitest only runs `src/**/*.test.ts` (`package.json` `test: vitest run`; no `scripts/*.test.ts` exist — the Story 3.2/5.2 thin-script precedent). So the reduced-view builder (Task 1) and the multi-stream ingest assembly (Task 2) live under `src/bundle/` where they are gate-tested; `scripts/build-bundle.ts` stays thin argv/fs/stdout glue. `src/bundle/` is the right home: it already holds `project-events.ts`/`classify-role.ts`/`assemble-bundle.ts` (the bundle-shaping layer) and is browser-UNREACHABLE (the script imports it; nothing on the `main.ts → render` path does). It imports `schema/` + the `ingest/` parsers + `classify-role` — all allowed for `src/bundle/` (the 5.5 precedent; no R1/R4/R5 crossing — it is SDK-free, phaser-free, and is NOT a Layer-0 module so it MAY import `ingest/` parsers, which `assemble-bundle`/`build-bundle` already do).

### §5 — `SNIPPET_MAX_CHARS` + the AC4 size bound (the budget)

- **`SNIPPET_MAX_CHARS = 200`** — a documented module `const`, NOT config-as-data. Rationale: this is a fixed prompt-budget engineering knob (how many chars per event the LLM needs to recognize a beat), not operator-tunable replay tuning (NFR-4 governs replay config — translation/pacing/captions — not a bake-input budget). Mirrors `classify-role.ts`'s inline-rules decision (Story 5.5 §3: "FIXED rules whose precedence is load-bearing logic, not operator-tunable data"). Keeping it inline avoids a config loader that earns nothing.
- **AC4 bound:** assert `Buffer.byteLength(JSON.stringify(view)) < 250_000` AND a token estimate `JSON.stringify(view).length / 4 < 60_000`. The chars/4 heuristic is the standard rough token estimate (documented as approximate, not exact — the real `claude -p` tokenizer differs, but the bound is conservative). With ~689 events × (eventId ~12 + toolName ~8 + role ~8 + outcome ~9 + snippet ≤200 + JSON overhead ~40) ≈ ~280 bytes/event worst-case ≈ ~193KB < 250KB, ≈ ~48K tokens < 60K. The full-events serialization (payloads up to ~28KB/event per the 5.5 §0 note) is materially larger — the test contrasts the two to prove the WHY. These are SAFE, conservative bounds proving a one-shot interpret fits the ~200K window with ample margin.

### §6 — Snippet extraction per normalized payload shape (verified against `normalize.ts`)

The reduced view runs over `NormalizedEvent[]` (post-ingest, post-scrub), whose `payload` is normalized (`normalize.ts` L82-112):
- `tool_use` → `payload = { input: <tool input object> }`. Bash: `input.command` (+ `input.description`). File ops: `input.file_path`/`input.path`/`input.content`/`input.old_string`/`input.new_string`.
- `tool_result` → `payload = { content: string | array-of-content-blocks | null }`. The array case (a real Claude Code shape, `normalize.ts` L40-53) has `{type:'text', text}` leaves — read the first text leaf.
- `prompt`/`text` → `payload = { text: string }`.
- `journal_started`/`journal_result` → `payload = { result, key }`; the short signal is the top-level `subtype` (status/verdict — `normalize.ts` L209-214), prefer that over digging into `payload.result`.
**Decision (snippet content):** extract the most tag-salient SHORT text per type (Task 1 list): Bash command head, tool_result pass/fail head, assistant/prompt text head, journal subtype, else `''`. **PATH-FREE:** do NOT echo full file paths into the snippet — the operator's Story 5.5 hard line is "no real file/path/symbol name in any retained surface." The reduced view is a TRANSIENT bake input (never shipped, never hashed, never logged by default — the CLI client references lengths/ids only, `claude-cli-client.ts` L126-145), so a path in the prompt is not a SHIPPED leak; but keeping snippets path-free is the consistent, defensible choice and prevents a future "log the prompt" change from leaking a path. The `role` token already conveys the file's KIND (test/schema/config/...) without the name. Test asserts `.not.toContain('/work/project')`. **Decision (scrub-safety, AC1/Hard-Invariant-3):** the builder runs on `scrubResult.scrubbedEvents` (AFTER Story 5.1 scrub — `build-bundle.ts` L124, L134), so any secret/PII in a snippet source was already redacted. The builder adds no un-scrubbing; it only truncates already-scrubbed text. Document that the builder must NEVER be called on raw/un-scrubbed events.

### §7 — The multi-stream CLI shape: BOTH repeated flags AND a manifest

**Decision:** support BOTH (a) repeated `--transcript <path> --stream-id <id>` pairs and (b) a `--session-manifest <path>` JSON (`[{transcript, streamId}, ...]`), with EXACTLY ONE supplied (fail-loud on both/neither). Rationale: the AC explicitly offers either; supporting both costs little (one small repeated-flag parser + one `.strict()` Zod loader) and serves two real ergonomics — repeated flags for a quick 2-transcript run, a manifest for the operator's documented full-session run (dev + fix transcripts + journal, where a checked-in manifest is the reproducible source of truth). The manifest loader is `.strict()` + Zod (fail-closed on a malformed table — the `scrub-patterns.ts` precedent). `--journal` stays a single required flag (one journal per session). The EXISTING single-`--transcript`/`--stream-id` invocation maps to a one-element `transcripts` array — UNCHANGED behavior (Hard Invariant 1). Put the manifest schema in a tiny `src/bundle/session-manifest.ts` leaf IF it earns a test; otherwise inline in the script (decide on merits — a `.strict()` Zod loader with a fail-closed test earns its keep, so prefer the leaf + a small test).

### §8 — Bake wiring: reduced view to the prompt, full events to provenance (AC3)

This is the load-bearing AC3 resolution. The interpreter does TWO things with its `events` arg: (1) serializes them into the prompt (`claude-interpreter.ts` L114), and (2) computes `sourceHash` over the grounded slice (`stampProvenance` L150-161 filters `events` by `groundingPointer.eventRefs`). AC3 requires the PROMPT to be the reduced view (so it fits) but `annotationHash` + provenance to be over the FULL scrubbed events.
- `annotationHash` is safe regardless: it is computed in `freezeAnnotations` (`assemble-bundle.ts` L74-79) over `scrubResult.scrubbedEvents` — the assembler is called with `scrubResult` UNCHANGED, so `annotationHash` is over the full events no matter what the interpreter prompt was. (Hard Invariant 2.)
- The PER-ANNOTATION `sourceHash` is the subtle one: if `interpret()` receives only the reduced view, `stampProvenance` would hash the grounded slice of the REDUCED view (which has no payload), not the full events. **Decision (preferred):** give `ClaudeInterpreter.interpret(promptEvents, groundingEvents = promptEvents)` an OPTIONAL second arg — `promptEvents` is serialized into the prompt; `groundingEvents` (the FULL scrubbed events) is what `stampProvenance` filters for `sourceHash`. Backward-compatible (default = same array → every existing single-arg caller and the `BeatInterpreter` interface contract are unaffected; the second param is a `ClaudeInterpreter`-private extension, NOT added to the `BeatInterpreter` interface — keep `interpret(events)` as the interface shape). The script calls `interpreter.interpret(taggingView, scrubResult.scrubbedEvents)`. This is the ONE `interpret/`-zone touch; R4 holds (the SDK import stays lazy, `claude-interpreter.ts` L167-173). **Saga:** `authorSaga(events)` only serializes (no provenance) — pass `taggingView` directly. The AC explicitly permits authoring the Saga over a reduced input ("it needs the arc, not bytes — dev's call, documented"); the reduced view IS that arc-bearing input (tool + role + outcome + snippet per event, in order), so reusing it for both is the simplest unified choice. Record: Saga input = reduced view (sufficient for the closing arc; the timeline/beats alternative buys nothing here since the view already carries the per-event narrative spine). The DEFAULT mocked path is UNTOUCHED (it never builds the view).

### §9 — Why this is safe + surgical (the inert-at-replay proof)

This story touches BAKE INPUT only. At replay nothing changes: the browser loads the same payload-free `projectedEvents` bundle, folds the same baked timeline, resolves the same abstracted grounding (Story 5.5). The reduced view + multi-stream ingest live entirely in the offline `scripts/` + `src/bundle/` path. `annotationHash`/freeze are over the full scrubbed events (unchanged). The committed fixture bundle is byte-identical (the mocked path never uses the reduced view; the single-transcript ingest reproduces the prior merge). BOTH golden snapshots are byte-stable (no Layer-0 code added). R1/R4/R5 hold (the new builders are SDK-free, phaser-free, and `src/bundle/`-resident, importing only `schema/`/`ingest/`-parsers/`classify-role` — the 5.5 precedent). The one `interpret/` touch (§8) is a backward-compatible optional param that keeps the SDK lazy.
