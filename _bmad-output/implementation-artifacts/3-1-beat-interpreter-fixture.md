# Story 3.1: BeatInterpreter interface + FixtureInterpreter overlay

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the builder,
I want a `BeatInterpreter` interface with a deterministic `FixtureInterpreter` double whose annotations apply as a read-only overlay,
so that the whole beat system is testable in CI with zero LLM calls and provenance discipline (R1) is enforced from the start. (FR-2 foundation)

## Acceptance Criteria

(Verbatim from `_bmad-output/planning-artifacts/epics.md#Story-3.1`, under "### Story 3.1: BeatInterpreter interface + FixtureInterpreter overlay".)

1. **Given** the `interpret/` module **When** the interface is defined **Then** `BeatInterpreter` produces `BeatAnnotation[]` and a `FixtureInterpreter` returns fixed annotations for the test fixture with no network call.
2. **Given** the annotations **When** they are consumed **Then** they apply as a READ-ONLY overlay — they never feed HP/pacing math, and no Layer-0 module imports `interpret/` (R1, enforced by lint) **And** every `BeatAnnotation` carries a `groundingPointer` resolving back to the Layer-0 event(s) it dramatizes.

### What the gate CAN prove (READ THIS — this story is FULLY gate-provable, unlike 2.3–2.5)

Unlike the renderer stories, Story 3.1 has **NO operator-verified surface** — there are no pixels, no felt interaction, no visual layout. Everything in both ACs is provable by the headless pipeline (`typecheck` / `lint` / `test` / `build`):

- **AC1** is proven by: (a) a `FixtureInterpreter` instance is assignable to the `BeatInterpreter` interface type (TS structural typing — a typecheck-level proof the double implements the contract); (b) `fixtureInterpreter.interpret(events)` returns a `BeatAnnotation[]` (each element `BeatAnnotationSchema.parse`-valid) with **no network/clock/RNG** (proven structurally — the module imports nothing from `node:*`, `@anthropic-ai/sdk`, or any I/O, and reads no `Date.now`/`Math.random`/`performance.now`; assert by source-grep in the test + by the test running offline in CI); (c) the returned set is **byte-identical across repeated calls** (determinism — call twice, `expect(a).toEqual(b)`).
- **AC2** is proven by: (a) **R1 is lint-enforced** — `eslint.config.ts` already forbids `ingest/translate/pace/model → interpret/` (committed in Story 1.1; a deliberate violation fails `pnpm lint`); the story adds NO Layer-0 import of `interpret/`, and a comment-level grep test asserts no Layer-0 file imports `interpret/`; (b) the **overlay is read-only** — `applyOverlay(timeline/events, annotations)` returns a NEW annotated view and the inputs are **structurally unchanged** (deep-equal the inputs to a pre-call snapshot — the exact `mergeStreams`-doesn't-mutate proof pattern from `ingest.test.ts` L127-138); the overlay touches NO `BattleState`/HP/`Beat.weight`/`dwellMs` field (there is no code path from the overlay into mechanics — assert the overlay's output type carries the annotations *alongside* the L0 data, never folded into it); (c) **every** `BeatAnnotation.groundingPointer.eventRefs` entry resolves to an `eventId` that EXISTS in the fixture's `NormalizedEvent[]` (no dangling refs — build a `Set` of the 14 fixture eventIds and assert every annotation's `eventRef` AND every `groundingPointer.eventRefs[i]` is a member).

There is nothing here a human must eyeball. If a test "can't" prove something, the design is wrong — fix the design, not the test. ## Dev Notes

### What this story IS (and is NOT)

- **IS:** the **Layer-1 ENTRY** for the signature-beat system (FR-2 foundation) — three small, pure, deterministic modules under `src/interpret/`: (1) `beat-interpreter.ts` = the `BeatInterpreter` INTERFACE (`interpret(events): Promise<BeatAnnotation[]>`); (2) `fixture-interpreter.ts` = a deterministic `FixtureInterpreter` test double returning FIXED hand-authored annotations for the committed ingest fixture with ZERO network/LLM/clock/RNG; (3) `overlay.ts` = a pure `applyOverlay` that pairs L0 events with L1 annotations into a READ-ONLY view (keyed by `eventRef`) for scribe/portal/render to consume — never mutating Layer-0. ### Overlay shape: side-by-side view + `byEventRef` index (NOT a fold into Layer-0)

`applyOverlay(events, annotations)` returns `AnnotatedView = { events: readonly NormalizedEvent[]; annotations: readonly BeatAnnotation[]; byEventRef: ReadonlyMap<string, readonly BeatAnnotation[]> }`. The design intent (architecture L386, L200-201): the portal/scribe/render overlay consume this to answer "what beat (if any) dramatizes the event at the cursor, and what does it ground to?" — so they need (a) the annotations, (b) a fast `eventId → annotations[]` lookup (`byEventRef`), and (c) the L0 events to resolve `groundingPointer.eventRefs` against. **Crucially the annotations live ALONGSIDE the events, NEVER folded into them** — no `event.annotation` field is written, no HP/weight is derived. This is the structural embodiment of R1: there is literally no code path from the overlay into `BattleState`/`Beat.weight`. The events pass through by reference (or shallow copy); `byEventRef` is the only new structure. Read-only `readonly` types + an optional `Object.freeze(view)` + the no-mutation test enforce it. (Alternative considered + rejected: returning `Beat[]` with attached annotations — rejected because it couples the overlay to the Pacer's `Beat` shape and tempts folding interpretation into the mechanics unit; keying off raw `eventId` keeps the overlay independent of pace/ and provably mechanics-free.) [Source: architecture.md#Architectural Boundaries L386, #Frontend Architecture L200-201, #R1 L225-228; src/schema/battle-timeline.ts L11-18 (Beat carries `sourceEventIds` — the same eventIds the overlay keys on, so a future portal can bridge beat→event→annotation), L37-45 (BattleState — the fields the overlay must NEVER carry); story-specific guidance "Returns an annotated VIEW, never mutates the inputs"]
