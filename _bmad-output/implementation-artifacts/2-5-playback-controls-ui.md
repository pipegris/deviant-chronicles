# Story 2.5: Playback controls UI

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a viewer,
I want on-screen playback controls wired to the cursor,
so that I can play, pause, restart, scrub, and change speed. (FR-6)

## Acceptance Criteria

(Verbatim from `_bmad-output/planning-artifacts/epics.md#Story-2.5`, L312-326.)

1. **Given** the Arena **When** I use the controls in `render/controls.ts` **Then** play, pause, restart behave predictably and a speed control (at least normal/fast) is available.
2. **Given** the scrub control **When** I drag to any point **Then** the Arena renders the correct state for that position (driven by the reducer cursor from Story 2.2).

### What the gate CAN prove vs what needs OPERATOR eyes (READ THIS — be honest)

The headless pipeline (`typecheck` / `lint` / `test` / `build`) **CAN** prove the *testable surface*: the **control→action WIRING** — that clicking the play/pause/restart buttons dispatches `{type:'play'|'pause'|'restart'}` to the reducer and the resulting `PlaybackState.status`/`cursor` update correctly; that moving the scrub control to value N dispatches `{type:'seek', cursor:N}` and the rendered `BattleState` equals `foldBattleState(timeline, N)` (the **scrub==play** equality, inherited from Story 2.2's path-independent reducer); that the speed toggle dispatches `{type:'setSpeed', speed:M}` and the per-tick advance multiplier changes (`tick` then advances M beats). It can also prove the **adapter is driven correctly** (a fake `RenderPort` records that `seek` triggers a SNAP `render(snapshot)` and a forward `tick` triggers `renderTransition(prev,next,beats)`), that the rAF loop **ticks only while `status==='playing'`** (a fake clock / injected stepper proves a paused loop advances 0 beats and a playing loop advances), R5 lint (`controls.ts` lives under `render/`), and `pnpm build`.

It **CANNOT** prove the **visual LAYOUT / styling / "feel"** — where the buttons sit, whether the scrubber is comfortable to drag, whether the speed toggle is legible, whether play/pause feels responsive at 60fps. AC1's "behave predictably" *as a felt interaction* and the control bar's appearance are **OPERATOR-verified** by running `pnpm dev` and clicking. The DOM-event→action wiring IS gate-provable under jsdom (dispatch a `click`/`input` event, assert the action fired and the state moved); the pixel placement and ergonomics are not. **Do NOT claim layout/feel the tests cannot show.** In the Completion Notes, state exactly what the wiring tests cover vs what needs human eyes. [Source: story-specific guidance "Split gate-verifiable (control→action wiring + scrub==play in jsdom) from operator-verified (visual layout/feel)"; architecture.md#Starter Template Evaluation L130-131 "Renderer excluded from determinism tests (timeline-level, not pixel-level)"; 2-4 story §"What the gate CAN prove"]

Senior-review fix round 1 (2026-06-15) — addresses every finding in the review record below.

Phaser DOM (`scene.add.dom`) would force every control test to boot a `Phaser.Game` under HEADLESS and reach into the scene — heavier, slower, and the HEADLESS DOM-plugin path is less reliable in jsdom. ### The scrub ↔ cursor mapping — RESOLVED on documented merits (the heart of AC2)
- **DECISION — the scrubber is a native `<input type="range">` with `min=0`, `max=timeline.beats.length`, `step=1`, and its `input` handler dispatches `{type:'seek', cursor: Number(slider.value)}`. NOTHING else.** The slider value IS the cursor (a beat index). No scaling, no percentage math, no interpolation.
- **Why this trivial map is CORRECT and sufficient:** Story 2.2 PROVED `seek(N).battleState === foldBattleState(timeline, N)` for EVERY integer N in `[0, beats.length]` — the every-cursor SCRUB==PLAY equality (its headline test). So "drag to any point → the Arena renders the correct state for that position" (AC2) reduces to "map the slider's value to a cursor and dispatch `seek`" — the reducer's path-independence does ALL the correctness work. The controls add ZERO state math; they cannot get the state wrong because they never compute it. [Source: src/model/playback.ts L96-113 (withCursor — every cursor-changing branch re-derives via foldBattleState, so seek==play by construction), L139-140; src/model/playback.test.ts L96-109 (the every-cursor proof); epics.md#Story-2.5 AC2 L324-326]
- **Why beat-index, not a percentage or a time(ms):**
  - **Beat-index matches the reducer's cursor domain EXACTLY.** `seek` takes a `cursor` (a beat index); the reducer floors + clamps it to `[0, beats.length]`. `step=1` on the range input means the slider only ever emits integer beat indices — no fractional seek to floor, no off-by-one at the ends. A percentage (0–100) or a time-in-ms slider would require the controls to convert (`pct/100 * beats.length` or a ms→beat lookup) — reintroducing exactly the state-math the trivial map avoids, and a rounding step that could desync `slider.value` from `state.cursor`. [Source: src/model/playback.ts L110-111 (floor + clamp to [0, beats.length]), L52 (seek carries `cursor` — a beat index), L60-69 (cursor/speed are integer beat counts); architecture.md#Frontend Architecture L193-195 "driven by a playback cursor"]
  - **`max = beats.length` (not `beats.length - 1`)** so the slider's far-right reaches the held-victory frame (`foldBattleState(timeline, beats.length)` = the final post-victory state; the reducer pins the cursor there). This matches the reducer's clamp upper bound exactly. [Source: src/model/playback.ts L111 (clamp(floored, 0, timeline.beats.length) — upper bound INCLUSIVE of beats.length); src/model/playback.test.ts L97-109 (the proof walks 0..beats.length inclusive)]
- **`input` vs `change` event:** use `input` (fires continuously DURING the drag) so the arena seeks LIVE as the user drags (the "drag to any point and watch" experience), not only on release. Each `input` dispatches a `seek`; the reducer's `seek` is idempotent + O(fold) and the SNAP render is cheap, so live scrubbing is safe without debounce in v0.1. (If profiling later shows fold cost on a long timeline, a render/-side debounce is the localized fix — NOT a reducer change.) [Source: src/model/playback.ts L136-140 (seek is a pure idempotent re-fold); MDN range input semantics: `input` fires during drag, `change` on commit]
- **Two-way reflection (sync), one-way data:** the slider's `.value` is also UPDATED by `sync()` when the cursor moves via `tick`/`restart` (so the thumb tracks forward playback and snaps to 0 on restart). But the slider only ever DISPATCHES `seek` on user `input`; `sync()` writing `slider.value` programmatically does NOT fire `input` (jsdom + browsers: setting `.value` in code does not dispatch `input`), so there is no feedback loop. The data flow is still one-way (controls→dispatch→reducer→render); the read-back is a UI reflection of the reducer's output. [Source: src/model/playback.ts L147-148 (tick moves the cursor — sync must follow it), L133-134 (restart → cursor 0); architecture.md#Communication Patterns L272-273; DOM spec: programmatic `.value` set does not fire `input`]

### Who renders — the controls or the boot — RESOLVED on merits
- **DECISION — the BOOT renders; the controls only dispatch + sync the UI.** `createControls` is handed a `dispatch(action)` callback. The BOOT supplies that callback as: `(action) => { state = reducer(state, action); afterDispatch(state, action); controls.sync(); }`, where `afterDispatch` does the SNAP render for cursor-jumps (`seek`/`restart`) via `adapter.render(state.battleState)`. The forward `tick` render stays in the rAF loop (animated path). The controls NEVER hold the adapter.
- **Why the boot, not the controls, calls the adapter:**
  1. **Single owner of state + adapter.** The boot already owns `let state`, the `reducer`, and the `adapter` (the rAF loop reassigns `state` and calls `adapter.renderTransition`). Routing the seek-render through the boot keeps ONE place that maps actions→renders — the controls don't need a second reference to the adapter (which would be a second edge that could drift from the loop's rendering). [Source: src/render/arena-boot.ts L51-87 (the boot owns state+reducer+adapter+loop)]
  2. **Keeps the controls renderer-agnostic + thin.** The controls speak only `PlaybackAction`/`PlaybackState` (types) + DOM — they have NO `RenderPort` dependency, so they are trivially unit-testable with a spy `dispatch` and would survive a renderer swap untouched. [Source: architecture.md#R5 L239-241; CLAUDE.md "Simplicity first"]
  3. **The snap-vs-animate decision lives with the action dispatcher.** Whether a render SNAPS (`render`) or ANIMATES (`renderTransition`) depends on whether the cursor JUMPED (seek/restart) or STEPPED (tick) — a distinction the boot already makes (it computes `beatsAdvanced` for ticks). Putting `afterDispatch` in the boot keeps that decision next to the loop that already embodies it. [Source: 2-4 story Dev Notes "animate forward, snap on seek"; src/render/arena-boot.ts L73-79 (the boot computes beatsAdvanced for the animate path)]
- **Alternative rejected — controls hold the adapter and render directly:** would give the controls a `RenderPort` dependency (no longer renderer-agnostic, harder to unit-test without a fake adapter, a second rendering edge). The recommended split is cleaner and matches the existing boot ownership. [Source: CLAUDE.md "Surgical changes"; src/render/arena-boot.ts (boot ownership)]
